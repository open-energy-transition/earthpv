Overview

A detector built to be reviewed, not trusted blind

Rooftop solar mapping in OpenStreetMap is dense in Germany and almost nonexistent in Pakistan, despite a rooftop PV boom driven by grid tariffs, load-shedding and net metering since 2022. earthpv closes that gap by fine-tuning TerraMind — a Sentinel-2/Sentinel-1 foundation model built by IBM Research and ESA (Φ-lab), served through TerraTorch — on Germany's labels, then running it over Punjab and, at country scale, all of Pakistan.

The design decision that follows from that gap: this is recall-first, deliberately. A false positive costs a human reviewer thirty seconds in an OSM validation tool (MapRoulette export included); a missed installation costs nothing but silently understates the true rooftop PV stock. Every stage — the training loss, the inference threshold, the postprocess ranking — is tuned to find candidates for review, not to hand down a verdict.

Architecture — process flow

Twelve CLI stages, chained through data/

Each stage is a Typer command that reads the previous stage's output files and writes its own. Nothing is a service — it's a pipeline of scripts over a shared data/ tree, run by hand or detached with nohup for the long ones.

1 · Labels

earthpv labels --aoi germany

Extract Overture buildings + OSM solar tags for an AOI — the ground truth chips are burned from. overpass-labels is the live-OSM alternative when Overture's snapshot lags reality.

ReadsOverture base/infrastructure + buildings (S3)
Writesdata/labels/<aoi>_solar.parquet
modules: labels.py · overture.py

2 · Chips

earthpv chips --aoi germany

Sample 224×224 px training chips and burn PV label masks. Positives are jittered ±900 m off-center — the fix for the tiling-artifact center bias (see Design invariants).

Readscomposites (local_source / compose) + labels
Writesdata/chips/<aoi>/{images,masks,index.parquet}
modules: chips.py · local_source.py · imagery.py

Hard-negative mining

earthpv hard-negatives hard-negative-chips

Confirms large, OSM-unmapped buildings as true negatives only if the model sees no PV signal in both the current composite and an older year's — bi-temporal agreement, not a single snapshot.

Readscomposites + checkpoint + VIDA + mapped refs
Writeshard_negatives_confirmed.parquet → new chips
modules: hard_negatives.py

Germany calibration

earthpv mastr calibrate

Aggregates the official MaStR PV registry per Gemeinde and zonal-joins it against model output — feeds a region-filter back into chip sampling and a kWp/m² calibration factor into density.

ReadsMaStR export + prob rasters + BKG VG250
Writeswell_mapped.parquet, calibration.json
modules: mastr.py · calibrate.py

3 · Train

earthpv train --config configs/terramind_pv.yaml

Fine-tunes terramind_v1_tiny via TerraTorch's SemanticSegmentationTask over a Lightning Trainer. --smoke runs 50 steps for a quick sanity check.

Readsconfigs/*.yaml + chip index.parquet
Writesdata/models/<run>/terramind-pv-*.ckpt
modules: train.py · datamodule.py · losses.py

4 · Evaluate

earthpv evaluate --aoi germany --checkpoint …

Pixel IoU/F1 on held-out chips, plus per-installation recall bucketed by array size — the number that actually matters for a recall-first detector.

Readschips index (val split) + checkpoint
Writes— (report only)
modules: evaluate.py

5 · Compose — AOIs without local imagery

earthpv compose --aoi pakistan --use-vida

Builds Sentinel-2 dry-season composites via Planetary Computer STAC, but only for building-populated 0.1° cells — rooftop PV needs roofs. Resumable, network-bound (~1–2 min/cell), prioritized by building density.

ReadsVIDA / local building points for cell selection
Writesdata/composites/<aoi>/composites/<cell>/composite_N.tif
modules: compose.py · imagery.py · buildings.py

6 · Infer

earthpv infer --aoi pakistan --checkpoint …

Slides a 224 px window (stride 104, deliberately not a multiple of 16) across each cell, Hann-blends the overlap into one seamless probability raster per cell.

Readscomposites + checkpoint
Writesdata/predictions/<aoi>/prob/<cell>.tif (uint8)
modules: infer.py · local_source.py

7 · Postprocess

earthpv postprocess --aoi pakistan --threshold 0.3

Thresholds and polygonizes, joins each candidate to the nearest building footprint (VIDA preferred), and computes rank_score = confidence × (0.5 + 0.5·building_prior). Nothing is dropped — the prior only reorders the review queue.

Readsprob rasters + VIDA/Overture buildings
Writesdata/predictions/<aoi>/candidates.parquet
modules: postprocess.py · buildings.py

8 · Export

earthpv export --aoi pakistan

Turns ranked candidates into human/OSM-consumable exports, ordered by rank_score — the queue a validator actually works through.

Readscandidates.parquet
Writes<aoi>_pv_candidates.{geoparquet,geojson}, _maproulette.geojson
modules: export.py

9 · Density — the energy-model export

earthpv density --aoi pakistan

Turns the same rasters into per-building PV density and 0.1°-grid / admin-region aggregates — PyPSA/PyPSA-Earth-ready, not a validation queue. No GPU, no retraining; resumable per-cell.

Readsprob rasters + candidates.parquet + VIDA
Writesdata/predictions/<aoi>/density/{buildings,grid,regions}.*
modules: density.py

The load-bearing design decision

Imagery and labels are reused, not re-downloaded

Sentinel-2 composites run to terabytes at country scale. Rather than refetch them, earthpv reads the Sentinel-2 composite COGs and OSM/Overture label + building parquets already produced by a sibling project, rooftopsenti, on the same drive — via local_root in configs/aoi.yaml.

rooftopsenti (sibling project)

Per-MGRS-tile Sentinel-2 composite COGs + OSM/Overture solar & building parquets, already downloaded for other work.

→

local_source.py

CompositeIndex, load_solar_labels, load_buildings — the primary imagery/label source for any AOI with a source_region.

Fallback — overture.py

DuckDB-over-S3 Overture Maps queries for labels/buildings/divisions when no local cache exists. Direct Overture S3 queries are slow from this machine — used sparingly.

Fallback — imagery.py

Planetary Computer / Earth Search STAC compositing for AOIs with no local composites at all (Punjab, Pakistan, Gujarat) — driven by the compose stage.

Consequence: an AOI is only fully usable where its source_region actually has composites. Punjab's assigned region (pakistan_500) covers Balochistan, not Punjab — so Punjab and country-wide Pakistan imagery is built on demand by compose, prioritized by building density since rooftop PV only exists where there are roofs.

How the estimate got here

1

Detected-area floor

Thresholded candidate polygons joined to footprints (est_mwp_det). Precision-honest, but blind to everything below the confidence threshold and the ~1,000 m² detection floor — i.e. blind to most residential PV.

2

Probability-weighted expectation

Σ per-pixel probability × 100 m² over each footprint (est_mwp_exp). Integrates sub-threshold signal; the two metrics bracket the truth from below and above.

3

Fraction-regression track

A second model head predicting per-pixel PV coverage fraction directly. Noisy per-installation (~4.5% recall at 0–250 m²) but unbiased in aggregate — chip-sum R² 0.60 on held-out Germany, municipal Spearman ρ 0.740 vs the legally-complete MaStR register (vs 0.499 for the segmentation baseline).

4

Calibration anchors

Germany: MaStR per-Gemeinde totals show a stable ~2.4–2.5× aggregate over-prediction (correctable). Pakistan: cross-checked against TransitionZero's 27.5 GW distributed-solar study, separating scale error inside imaged cells from cells never imaged at all.

5

Coverage expansion

That comparison showed missing coverage, not scale error, dominated: the local Overture ≥500 m² building set undercounts small/informal structures by 200–1,000× in rural Pakistan. Switching cell selection to VIDA Open Buildings (76.5 M footprints) grew the compose target from 122 to ~4,460 cells.

6

Next — the OSM flywheel

Validated leads feed back into OSM as in-domain Pakistani training positives (a retrain is pending); NEPRA net-metering totals as a Pakistani MaStR analogue; two-epoch ΔMWp (below) turns the estimate into a time series of the 2022–2026 boom itself.

Do not regress these

Two fixes that killed a grid of false positives

Naive sliding-window inference produced a regular grid of false positives at the window stride spacing — 60% of detections landed exactly one stride apart. Two invariants fixed it and must stay in place.

window 1 — 224 px
window 2
window 3
stride 104 px — not a multiple of the 16 px ViT patch
Training

Jittered positive chips

Positive chips are sampled with the installation off-center by up to ±900 m (chips.py::sample_chip_centers). Without jitter the model learns "PV sits in the middle of the frame" and fires once per window at inference — a grid at the stride spacing. This was the dominant cause: after the fix, one-stride-apart detections dropped from 60% to ~7%.

Inference

Hann-tapered overlap-add, non-aligned stride

infer.py blends overlapping windows with a 2D Hann taper into one seamless raster per cell, using a stride (104 px) that is deliberately not a multiple of the 16 px ViT patch size, so patch-edge effects decorrelate between neighbouring windows instead of reinforcing each other.

What's next

Three levers on the false-positive rate

Planned

Two-epoch change detection

Almost every real Pakistani rooftop array visible today didn't exist before the post-2022 solar boom. Composite a pre-boom epoch (2021-10 → 2022-01-24, same grid), run the unchanged model on both, and difference the probability surfaces. Persistent false positives — riverbeds, outcrops, industrial roofs — fire in both epochs and cancel; only new PV survives. No retraining required.

Planned

Sentinel-1 as a post-hoc filter

TerraMind ships pretrained S1 embeddings. Rather than a full retrain, multi-temporal backscatter variance at candidate locations cheaply separates permanent structures from seasonally-changing fields and flags greenhouses (metal-frame corner reflectors — bright, the opposite of PV's flat return).

In progress

Hard-negative mining loop

MapRoulette-rejected candidates are confirmed, in-domain hard negatives. hard_negatives.py already mines bi-temporal-confirmed negatives from large unmapped buildings; the same loop closes over human review rejections, jittered and oversampled like positives so the model doesn't re-learn a center bias in reverse.

Try it

Setup & the full pipeline

Managed with pixi, two environments in one solve-group: default (the data pipeline — DuckDB, geopandas, rasterio, no PyTorch) and ml (+ torch/torchvision cu126 + terratorch, for the GTX 1060's Pascal architecture).

pixi install            # default env — data pipeline
pixi install -e ml       # + PyTorch cu126 + TerraTorch
pixi run -e ml gpu-check # verify torch.cuda + device name
pixi run earthpv labels --aoi germany
pixi run earthpv chips --aoi germany
pixi run -e ml earthpv train --config configs/terramind_pv.yaml
pixi run -e ml earthpv evaluate --aoi germany --checkpoint data/models/<best>.ckpt

# Punjab/Pakistan have no local imagery — build it on demand, resumable
pixi run -e ml earthpv compose --aoi pakistan --min-buildings 1000 --use-vida
pixi run -e ml earthpv infer --aoi pakistan --checkpoint data/models/<best>.ckpt

pixi run earthpv postprocess --aoi pakistan --threshold 0.3
pixi run earthpv export --aoi pakistan
pixi run earthpv density --aoi pakistan
